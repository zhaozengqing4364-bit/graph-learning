## Problem Statement

AxonClone 的第一击不应该是一个完整的长期学习操作系统，而应该是一个足够锋利的最小体验:

用户扔进一篇复杂技术长文后，系统可以自动抽出 5 到 8 个关键概念，标出主线和前置关系，允许用户点开概念看到一句话解释和 "why now"，并直接给出一个简单明确的学习顺序，例如 "先学这 3 个，再回来看原文"。

这个体验面向的不是泛学习用户，而是两类高度重合的人:

- 需要把复杂技术讲清楚的学生或工程师
- 经常看长文章，但看完讲不出来的人

对这类人来说，真正的问题不是"找不到解释"，而是"读完以后仍然无法组织理解，更无法说出口"。

## What Makes This Cool

最酷的地方不是图本身，而是文章被压成可执行学习路径的那一刻:

- 一篇难文章不再只是被总结，而是被长成一张可学习的地图
- 图里不是平铺概念，而是能看出主线和前置关系
- 系统不是只说"这篇文章讲了什么"，而是直接说"你应该先学什么，为什么"
- 用户第一次就能感受到: 这不是问答器，也不是总结器，而是一个把复杂内容变成学习顺序的工具

长期理想形态已经很清楚，但不应成为第一版范围:

- 它会自己长出一门"微课程"，不只是一个图
- 它能发现你哪里只是看懂了，其实讲不出来

## Constraints

- 固定技术栈不变: Tauri v2、React + TypeScript + Vite、Tailwind、FastAPI、OpenAI Responses API、Neo4j、LanceDB、SQLite
- 当前仓库已经有 article-first workspace、graph、node detail、practice、summary、review 等较完整骨架，优先复用，不重做架构
- 第一楔子必须在 5 分钟内让首次用户感知价值，不能依赖长期留存
- 第一版可以明确收窄到"技术长文"场景，不需要一开始支持所有内容类型
- 当前项目真实状态说明工程基础还需收敛: `npm run lint` 失败，`npm run build` 失败，后端测试 `4 failed, 209 passed`
- 第一阶段明确牺牲长期沉淀和复习，把"文章 -> 地图 -> 学习顺序"做顺

## Premises

1. 这个项目的第一产品价值，不是"陪你长期学习"，而是"把一篇难技术文章压缩成一个可执行的学习地图"。
2. 第一波用户不是泛学习人群，而是"需要把复杂技术讲清楚的学生或工程师"。
3. 第一版最应该追求的，不是图谱有多复杂，而是用户在 5 分钟内看到"先学哪 3 个概念，为什么"的清晰路径。

## Approaches Considered

### Approach A: Article-to-Map First

先只做最强主流程: 长文章输入后，自动抽出 5 到 8 个关键概念，标出主线和前置关系，给出"先学这 3 个"的阅读顺序，点开概念能看到一句话解释和 why now。练习、复习、长期资产全部降到第二阶段。

- Effort: M
- Risk: Low
- Pros:
  - 最贴合项目当前想证明的第一瞬间价值
  - 能最大化复用现有 article workspace、entry node、graph 和 node detail 能力
  - 最容易做成一个 5 分钟内就能看懂和分享的 demo
- Cons:
  - "讲不出来"这一半价值暂时只停留在产品叙事里，还没被功能证明
  - 如果图谱质量一般，容易被误解为"更花哨的总结工具"
  - 长期学习 OS 的完整形象会先被延后
- Reuses:
  - 现有 `article-workspace-page`
  - 现有 graph / node detail / entry node / topic APIs
  - 现有文章阅读与概念展开交互

### Approach B: Guided Micro-Course

不把图当主角，而是把文章自动长成 3 到 5 步的微课程。默认界面不是图，而是"先学 A，再学 B，再回原文"的顺序化学习卡片，图谱退到辅助层。

- Effort: M/L
- Risk: Medium
- Pros:
  - 更接近项目 10x 愿景，不只是图，而是真正的课程化体验
  - 用户更容易直接行动，不会在图中迷路
  - 更符合"帮助用户讲清楚复杂技术"的目标
- Cons:
  - 第一版产品表达会更复杂，容易让改动面重新变大
  - 需要重新定义图谱、阅读器、路径卡片的主次关系
  - 如果路径质量不稳定，失败成本比图更高
- Reuses:
  - 现有 article workspace
  - 现有 session flow 与推荐逻辑
  - 现有 node/topic service 里的解释和排序能力

### Approach C: Map Then Speak

先做 A，但在用户点开第一个概念后，立即插入一个极轻的表达动作: "用自己的话讲一下这个概念"，系统给出单轮反馈，只暴露"讲不出来"这件事，不做完整长期练习和复习。

- Effort: M/L
- Risk: Medium
- Pros:
  - 更快体现 AxonClone 与 NotebookLM 的本质区别
  - 地图和表达两张牌都会在第一次体验中露出来
  - 不需要先把完整学习 OS 全做完
- Cons:
  - 第一版范围容易再次膨胀
  - 如果表达反馈不够准，会稀释 Article-to-Map 的主惊喜
  - 对 prompt 和 session 上下文质量要求更高
- Reuses:
  - Approach A 的全部基础
  - 现有 practice 和 diagnoser 相关服务

## Recommended Approach

选择 **Approach A: Article-to-Map First**。

原因不是它最保守，而是它最能准确证明项目的第一楔子: 用户把一篇难技术文章扔进去，系统真的能把它压成一张可学习地图，并明确告诉用户先学什么。

这一步一旦成立，AxonClone 就已经和纯总结、纯问答、纯知识图工具分开了。之后再接上 "讲出来" 和 "长期沉淀"，是顺势延长闭环，而不是同时赌三个价值点。

## Open Questions

1. 第一版如何定义"关键概念"质量，避免抽出来的是词，不是可学对象？
2. 主线和前置关系如何向用户解释可信度，避免看起来像随意连线？
3. "先学这 3 个" 应该基于图结构、文章顺序，还是 why_now 强度综合排序？
4. 第一版默认主视图应该是图谱、列表，还是一张"推荐学习顺序卡"？
5. 是否需要明确加入一组 demo 文章作为展示样本，保证首次体验质量？

## Success Criteria

- 用户粘贴一篇复杂技术长文后，系统能稳定抽出 5 到 8 个关键概念
- 系统能区分主线概念和前置概念，而不是只生成一堆并列节点
- 用户点开概念时，能看到一句话解释和"为什么现在学它"
- 系统能输出一个非常明确的建议，例如"先学这 3 个，再回原文"
- 首次体验者在 5 分钟内能说出: "这篇文章我应该怎么学"
- 用户不会把它误解成普通 AI 总结器，而能感知到"这是可学习路径生成器"

## Next Steps

1. 收窄输入范围，先定义第一版只服务"复杂技术长文"。
2. 把当前项目首页和 article workspace 收束成一条单一 demo 路径: 输入文章 -> 生成地图 -> 查看推荐顺序。
3. 为"关键概念"、"主线"、"前置"建立最小可解释规则，避免生成结果像黑盒。
4. 在当前工作台里把概念详情卡统一成两件事: 一句话解释 + why now。
5. 把"先学这 3 个"作为第一版最重要的输出，而不是附属提示。
6. 暂时下调复习、资产、长期沉淀的实现优先级，避免第一版分心。
7. 在第一楔子跑顺后，再把"讲出来并指出卡点"接回第二阶段。

## What I Noticed About How You Think

- 你没有把项目描述成泛泛的"AI 学习工具"，你直接说了: "一篇长文章扔进去，系统自动长成可学习的知识地图。"
- 你第二个 instinct 不是继续堆图，而是马上补了另一半价值: "学完一个概念后，系统逼你讲清楚，并指出你到底卡在哪。"
- 你对目标用户的判断很具体，不是"所有学习者"，而是"需要把复杂技术讲清楚的学生或工程师"和"经常看长文章，但看完讲不出来的人"。
- 你愿意主动收范围，这一点很重要。你明确选了: "先不做长期沉淀和复习，只把‘文章 -> 地图 -> 学习顺序’做得很顺。"
